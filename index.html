· How vetlock works

Six stages, one never-execute invariant. vetlock reads two lockfiles — the one you shipped and the one you're about to ship — and walks the dependency delta without running a single line of package code. What you see below is the actual pipeline: diff, changed-set, recursive fetch, detectors, provenance graph, verdict.

  1. stage 01 · diff

    Lockfile diff

    Two package-lock.json files in, one structural delta out. vetlock parses each lockfile deterministically and emits added, removed, and version-changed entries — no network, no code, just the tree of what actually shipped.

  2. stage 02 · changed set

    Changed set

    The delta collapses into a small set of package@version pairs to investigate. Everything else in the tree is evidence of what didn't move and gets skipped — a scan is only ever as expensive as the change is wide.

  3. stage 03 · recursive fetch

    Recursive fetch

    For each changed root, vetlock walks its full transitive closure via the npm registry — HMAC-authed cache in front, so a real repo scan hits the network once and reuses the tarballs across every subsequent detector. Nothing is executed; tarballs are only parsed.

  4. stage 04 · detectors

    Detectors run

    Seventeen static detectors pass over each tarball: install-script shape, new endpoints, env-token harvest patterns, obfuscation deltas, integrity tampering, typosquat neighbors, and more. Every detector emits evidence — file, line, capability. Never a claim without a citation.

  5. stage 05 · provenance

    Provenance graph

    Findings alone aren't enough — every node in the changed closure carries its origin, integrity, and adjacency: which registry published it, which signatures match, and how close it lies to a known advisory or typosquat cluster. Provenance is what turns a detector hit into a decision.

  6. stage 06 · verdict

    Verdict

    A per-package severity (CLEAN, WARN, BLOCK) and a whole-scan exit code — enough for CI to fail the merge automatically, with the evidence attached. The scan finishes with the same never-execute guarantee it started with: zero package code was run.

An observability platform for enterprise operations telemetry. Zero-touch delivery

Bump PR → policy-gated approval → auto-merge → drift detection → pin-back rollback.

Merge to main and walk away — the loop lands the change, watches for drift, and pins back to the last-known-good commit without a human in the middle.

Read the design

A CI bot authors image-bump PRs, a policy-gated approval clears them, auto-merge lands them, a drift detector watches for divergence between observed and desired state, and a pin-back rollback returns any regression to its last-known-good commit — end-to-end, no human in the middle.

The design constraint: the loop preserves the GitHub rails that keep humans safe (a PR can't approve itself; the author can't merge their own change) while still running end-to-end without a human in the middle — a constraint that quietly breaks naive deploy automation.

· Red team public report

I hired six adversaries to break my own security tool.

Six specialised red-team agents, one brief: find a way past vetlock. They proposed 48 candidate exploits across six territories — AST-level evasions, lockfile parsing blindspots, cache and config trust, multi-primitive compositions, defender-corner moves that turn vetlock's own configuration against it, and orthogonal blindspots nobody had thought about. A separate adversarial verifier agent triaged each candidate with a default-refuted stance: no proof-of-concept, no confirmation.

After dedup, 35 exploits were confirmed against the v0.2.0 baseline — a 73% confirmation rate. Two were refuted, one plausible, seven partial. Every finding is catalogued with a stable ID, a proof-of-concept, and a fix-or-accept decision in the public red-team report ↗.

As of v0.3.0 plus a tail audit, 37 of 48 catalogued exploits are closed, each with a dedicated REDTEAM <ID> FIX code tag in the codebase. Eleven remain open — deliberately, and the report names them. The three cheapest, highest-impact avenues are documented so the next reviewer knows exactly where to push.

  1. L

    Language — AST pattern evasions

    see report ↗

    Constant-folding gaps, alias-tracker corner cases, dynamic require shapes. L9 (the Shai-Hulud shape) was the anchor case at v0.2.0 — an OBF + EXEC + ENV + FS chain that stayed WARN because OBF escalation only fired on NET or INSTALL. Closed in v0.3.0 by broadening Escalation 1 in packages/detectors/src/index.ts to promote OBF → BLOCK on any of NET / INSTALL / EXEC / ENV / FS in the same package.

    IDs L1–L11

  2. F

    Format — lockfile parsing blindspots

    see report ↗

    npm-only assumptions vs. pnpm and yarn lockfile shapes. F1 was the report's anchor example at v0.2.0 — a pnpm/yarn shape that crashed the scanner to exit 3, which most CI configurations treated as an infra blip. Closed in v0.3.0; see the report for the timeline.

    IDs F1–F3, F5–F10

  3. S

    System — cache, TOCTOU, config trust

    see report ↗

    HMAC-authenticated cache holds up; the softer surface is trust in on-disk config. S2 — a malicious .vetlock.json shipped in the same PR — is the report's cheapest silencer.

    IDs S1–S10

  4. C

    Compositional — multi-primitive chains

    see report ↗

    Chains that combine two individually-benign primitives to escape a single-signal escalation rule. Fixes gated on a rework of the escalation graph, called out in the report.

    IDs C1–C6

  5. D

    Defender-corner — self-exemption

    see report ↗

    Config-based moves that turn vetlock's own knobs against it — allowlist an attacker's package, disable a category, silence via a shipped config. Fixes are deferred to a signed-config redesign called out in the report.

    IDs D1–D6

  6. N

    Novel — orthogonal blindspots

    see report ↗

    The wildcard bucket. N1 (bundledDependencies bypass — packages the lockfile never lists because they ship inside another package) is the report's single highest-impact open finding.

    IDs N1–N6

Per-ID dispositions (closed / open) are enumerated in the full report ↗.

05 Writing

  • The vetlock red-team review

    Six adversarial agents, 48 exploit candidates, an adversarial verifier that defaulted to REFUTED — 35 confirmed end-to-end. 39 of 48 catalogued exploits closed as of v0.4.2. What the numbers said the tool wasn't yet.

  • When correctness beats performance

    A 3x-faster string-distance library looked like an easy win — until "attackers rarely transpose" collided with npm's own typosquat history. Reverted, with tests that pin every transposition case in place — the principle that governs every security-boundary choice in vetlock.

  • Self-building instrumentation

    Hand-written analytics loses to features. Auto-capture breaks on every redesign. Nothing captures what a click was for. A sense-understand-act loop that names the UI from its own witnessed effects — auto-labeled at capture across a federated micro-frontend estate.

  • The two-identity problem in zero-touch delivery

    Two GitHub rules — no self-approval, and GITHUB_TOKEN events don't cascade — quietly break naive deploy automation: the PR opens, required checks never fire, auto-merge stalls forever. Four arrangements that survive them.
